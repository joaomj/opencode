# Codebase Diagrams

Generate Excalidraw diagrams for codebase onboarding and architecture explanations.

## When to Use This Skill

Use this skill when:

- Creating or updating `docs/tech-context.md`
- User requests: "diagram the architecture", "show data flow", "create sequence diagram"
- Explaining system structure to new team members
- Visualizing component relationships

## Supported Diagram Types

| Type | Best For | Max Elements |
|------|----------|--------------|
| **Architecture** | System components and their relationships | 12 |
| **Data Flow** | How data moves through the system | 10 |
| **Sequence** | API/service interactions over time | 8 |

## Diagram Selection Guide

| User Request | Diagram Type |
|--------------|--------------|
| "Show the system architecture" | Architecture |
| "How does data flow through the app?" | Data Flow |
| "Explain the API interactions" | Sequence |
| "What are the main components?" | Architecture |
| "How is data processed?" | Data Flow |
| "Show the request lifecycle" | Sequence |

## Prerequisites

- Clear understanding of codebase structure
- Entry points identified
- Major components defined
- Data flow paths understood

## Step-by-Step Workflow

### Step 1: Analyze Codebase

Before generating, identify:

**For Architecture Diagrams:**
- Main modules/components
- Entry points (API routes, main functions)
- Dependencies between components
- External services/APIs

**For Data Flow Diagrams:**
- Data sources (databases, APIs, files)
- Processing steps (transformations, validations)
- Data destinations (databases, APIs, UI)
- Direction of data movement

**For Sequence Diagrams:**
- Actors/entities involved
- Message order
- Synchronous vs asynchronous calls
- Response handling

### Step 2: Extract Structure

**Architecture Mapping:**
```
Components: API Gateway, Auth Service, User Service, Database
Connections: API Gateway → Auth Service → Database
             API Gateway → User Service → Database
```

**Data Flow Mapping:**
```
Sources: Client Request, Database
Steps: Validate → Transform → Store
Destinations: Database, Response
```

**Sequence Mapping:**
```
1. Client → API: POST /users
2. API → Auth: validate_token()
3. Auth → API: token_valid
4. API → UserService: create_user()
5. UserService → Database: INSERT
6. Database → UserService: user_id
7. UserService → API: User object
8. API → Client: 201 Created
```

### Step 3: Generate Excalidraw JSON

**Element Guidelines:**

| Element | Use Case | Style |
|---------|----------|-------|
| Rectangle | Components, services, databases | Light blue (`#a5d8ff`) |
| Diamond | Decision points (sequence only) | Light yellow (`#ffd43b`) |
| Ellipse | External systems, entry points | Light green (`#b2f2bb`) |
| Arrow | Connections, data flow | Gray (`#495057`) |
| Text | Labels, descriptions | Black, 16-20px |

**Font Requirements:**
- Always use `fontFamily: 5` (Excalifont) for all text
- Font size: 16-20px for labels
- Line height: 1.25

**Spacing:**
- Horizontal gap: 200-250px between components
- Vertical gap: 100-150px between layers
- Consistent padding within containers

**Colors:**
```
Primary (components): #a5d8ff (light blue)
Secondary (external): #b2f2bb (light green)
Important (entry): #ffd43b (yellow)
Alerts/Errors: #ffc9c9 (light red)
Text: #000000 (black)
Arrows: #495057 (gray)
Background: #ffffff (white)
```

### Step 4: Create Output File

**File Format:**
```json
{
  "type": "excalidraw",
  "version": 2,
  "source": "opencode-codebase-diagrams",
  "elements": [
    // Diagram elements
  ],
  "appState": {
    "viewBackgroundColor": "#ffffff",
    "gridSize": 20
  },
  "files": {}
}
```

**Naming Convention:**
- Architecture: `docs/diagrams/<project-name>-architecture.excalidraw`
- Data Flow: `docs/diagrams/<project-name>-data-flow.excalidraw`
- Sequence: `docs/diagrams/<scenario>-sequence.excalidraw`

### Step 5: Update Documentation

After generating, add reference in `docs/tech-context.md`:

```markdown
## System Architecture

![Architecture Diagram](./diagrams/<project>-architecture.excalidraw)

*Generated by codebase-diagrams skill*
```

## Architecture Diagram Structure

### Layout Pattern (Top to Bottom)

```
┌─────────────────────────────────────────┐
│          Client/UI Layer                │  ← Top (entry points)
└─────────────────┬───────────────────────┘
                  │
┌─────────────────▼───────────────────────┐
│          API Gateway Layer              │
└─────────────────┬───────────────────────┘
                  │
┌─────────────────▼───────────────────────┐
│          Service Layer                  │  ← Middle (business logic)
│  ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐       │
│  │Svc 1│ │Svc 2│ │Svc 3│ │Svc 4│       │
│  └─────┘ └─────┘ └─────┘ └─────┘       │
└─────────────────┬───────────────────────┘
                  │
┌─────────────────▼───────────────────────┐
│          Data Layer                     │  ← Bottom (databases)
│  ┌────────┐ ┌────────┐ ┌────────┐      │
│  │  DB 1  │ │  DB 2  │ │Cache   │      │
│  └────────┘ └────────┘ └────────┘      │
└─────────────────────────────────────────┘
```

## Data Flow Diagram Structure

### Layout Pattern (Left to Right)

```
┌──────────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐
│  Source  │────▶│ Process  │────▶│ Process  │────▶│   Sink   │
│  (Data)  │     │  Step 1  │     │  Step 2  │     │  (Save)  │
└──────────┘     └──────────┘     └──────────┘     └──────────┘
```

### Key Elements

- **External Entities** (ellipses): Sources and destinations
- **Processes** (rectangles): Data transformations
- **Data Stores** (rectangles with double borders): Databases, caches
- **Data Flows** (arrows with labels): Direction and data type

## Sequence Diagram Structure

### Layout Pattern (Time flows top to bottom)

```
Client          API          Auth        Database
   │             │             │             │
   │─────────────▶             │             │  Request
   │             │─────────────▶             │  Validate
   │             │◀─────────────│             │  Valid
   │             │                           │  Query
   │             │───────────────────────────▶│
   │             │◀───────────────────────────│  Result
   │◀────────────│                           │  Response
   │             │             │             │
```

### Elements

- **Lifelines**: Vertical dashed lines for each entity
- **Activation Boxes**: Rectangles on lifelines during execution
- **Messages**: Horizontal arrows (solid = sync, dashed = async)
- **Return Messages**: Dashed arrows with return values

## Best Practices

### Element Limits

| Diagram Type | Recommended | Maximum |
|--------------|-------------|---------|
| Architecture | 8-10 | 12 |
| Data Flow | 6-8 | 10 |
| Sequence | 6-8 | 8 |

**If exceeding limits:**
- Create multiple focused diagrams
- Show high-level overview + detailed sub-diagrams
- Use layers/hiding in Excalidraw

### Layout Tips

1. **Grid Alignment**: Use 20px grid, align to grid
2. **Consistent Sizing**: Same-size components where possible
3. **Label Positioning**: Above rectangles, beside arrows
4. **Color Coding**: Consistent palette per diagram type
5. **Whitespace**: Leave 50-100px margin around diagram

### Text Guidelines

- **Component Names**: Title case, 1-2 words
- **Labels**: Short, descriptive (max 4-5 words)
- **Font Size**: 18px for titles, 16px for labels
- **Text Alignment**: Center in shapes, left on arrows

## Troubleshooting

| Issue | Solution |
|-------|----------|
| Elements overlap | Increase spacing, reduce element count |
| Text too small | Increase font size to 20px |
| Arrows unclear | Add labels, use different arrow styles |
| Too many elements | Create multiple focused diagrams |
| Colors inconsistent | Use defined palette only |

## Limitations

- Maximum 15 elements per diagram (complexity threshold)
- Basic shapes only (rectangles, ellipses, diamonds, arrows)
- No icon libraries (basic shapes preferred for portability)
- No auto-layout (manual positioning)
- Static diagrams only (no animations)
